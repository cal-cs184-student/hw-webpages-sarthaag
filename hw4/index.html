Part 3: Handling collisions with other objects

For Part 3, I implemented collision handling for spheres and planes and added collision checks in the cloth simulation loop. For sphere collisions, I implemented Sphere::collide by checking whether a point mass is inside or intersecting the sphere using the distance from the sphere center. If it is inside, I compute the tangent point on the sphere surface along the direction from the sphere center to the point mass. I then compute a correction vector from last position to this tangent point. The final updated position is last position plus the correction scaled by 1 minus friction. This pushes the point mass back to the sphere surface and applies friction. For plane collisions, I implemented Plane::collide by checking if the point mass crossed the plane between last position and current position. I do this by checking the sign of the dot product with the plane normal at both positions. If the signs differ, the point crossed the plane. I compute the tangent point using line plane intersection along the path from last position to current position. I then move this tangent point slightly above the plane using SURFACE_OFFSET on the same side where the point mass came from. I compute the correction vector from last position to that adjusted point, then set the new position to last position plus the correction scaled by 1 minus friction. In Cloth::simulate, I added a loop so every point mass is tested against every collision object each time step. This applies sphere and plane collision response during simulation. For scene sphere json, I compared final resting results for ks = 500, 5000, and 50000. With ks = 500, the cloth is softer and stretches more, so it sags deeply and wraps more around the sphere. With ks = 5000, the cloth drapes naturally with moderate stretch. With ks = 50000, the cloth is stiffer, keeps more structure, and sags less over the sphere. For scene plane json, the cloth falls onto the plane and rests on the surface without clipping through it. The small surface offset helps keep the cloth just above the plane for stable contact.

Part 4: Handling self-collisions

To efficiently handle cloth self-collisions without incurring a performance penalty, our implementation utilizes spatial hashing to divide the 3D space into discrete grid volumes. During each simulation step, a hash map is dynamically built to group point masses by their 3D grid coordinates, which are converted into unique 1D identifiers using a prime-number-based hash function. Then, for every point mass, collisions are only checked against other candidate masses within that exact same spatial box. If two point masses are found to be closer than twice the cloth's thickness, a corrective vector is computed to push them apart. Finally, all corrections for a given point mass are averaged and added to its position, scaled down by the total number of simulation steps to ensure stable, jitter-free collision resolution.

Part 5: Shaders

A shader program is a piece of software designed to run directly on a GPU and built for massive parallel processing, allowing for the calculation of the positions and colors of millions of pixels simultaneously. Vertex shaders calculate where an object is and how it's oriented towards light, and it hands that information off to the fragment shader which takes that data plus the light sources plus the camera angle and computes the exact color that each pixel should be.

The Blinn-Phong shading model is a formula to simulate how light interacts with surfaces that gives object a shiny or metallic appearance and is computationally cheap. It breaks down light into ambient light, diffuse light, and specular light and adds them together.
