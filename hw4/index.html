Part 3: Handling collisions with other objects

For Part 3, I implemented collision handling for spheres and planes and added collision checks in the cloth simulation loop. For sphere collisions, I implemented Sphere::collide by checking whether a point mass is inside or intersecting the sphere using the distance from the sphere center. If it is inside, I compute the tangent point on the sphere surface along the direction from the sphere center to the point mass. I then compute a correction vector from last position to this tangent point. The final updated position is last position plus the correction scaled by 1 minus friction. This pushes the point mass back to the sphere surface and applies friction. For plane collisions, I implemented Plane::collide by checking if the point mass crossed the plane between last position and current position. I do this by checking the sign of the dot product with the plane normal at both positions. If the signs differ, the point crossed the plane. I compute the tangent point using line plane intersection along the path from last position to current position. I then move this tangent point slightly above the plane using SURFACE_OFFSET on the same side where the point mass came from. I compute the correction vector from last position to that adjusted point, then set the new position to last position plus the correction scaled by 1 minus friction. In Cloth::simulate, I added a loop so every point mass is tested against every collision object each time step. This applies sphere and plane collision response during simulation. For scene sphere json, I compared final resting results for ks = 500, 5000, and 50000. With ks = 500, the cloth is softer and stretches more, so it sags deeply and wraps more around the sphere. With ks = 5000, the cloth drapes naturally with moderate stretch. With ks = 50000, the cloth is stiffer, keeps more structure, and sags less over the sphere. Part 4: Handling self-collisions

To efficiently handle cloth self-collisions without incurring a performance penalty, our implementation utilizes spatial hashing to divide the 3D space into discrete grid volumes. During each simulation step, a hash map is dynamically built to group point masses by their 3D grid coordinates, which are converted into unique 1D identifiers using a prime-number-based hash function. Then, for every point mass, collisions are only checked against other candidate masses within that exact same spatial box. If two point masses are found to be closer than twice the cloth's thickness, a corrective vector is computed to push them apart. Finally, all corrections for a given point mass are averaged and added to its position, scaled down by the total number of simulation steps to ensure stable, jitter-free collision resolution.

Part 5: Shaders

A shader program is a piece of software designed to run directly on a GPU and built for massive parallel processing, allowing for the calculation of the positions and colors of millions of pixels simultaneously. Vertex shaders calculate where an object is and how it's oriented towards light, and it hands that information off to the fragment shader which takes that data plus the light sources plus the camera angle and computes the exact color that each pixel should be.

The Blinn-Phong shading model is a formula to simulate how light interacts with surfaces that gives object a shiny or metallic appearance and is computationally cheap. It breaks down light into ambient light, diffuse light, and specular light and adds them together.

Bump mapping creates an optical illusion -- it only modifies the normal vectors during the fragment shader phase to fake how light hits the surface. It does not alter the physical 3D mesh. Displacement mapping actually modifies the geometry -- it reads the height map in the vertex shader and pushes the vertices outward or inward along their normals before applying the lighting calculations. In the bump mapping render, the surface of the sphere looks deeply textured and bumpy, but it remains perfectly round and smooth when one looks at its outer silhouette against the background. In contrast, the displacement mapping render shows the physical geometry has changed. The silhouette of the sphere is heavily jagged and deformed, accurately reflecting the physical peaks and valleys of the brick texture.

For bump mapping, rendering with -o 16 -a 16 yields a low-polygon outline of the sphere, but the flat faces of those blocks on the sphere still display a highly detailed, crisp bumpy texture. Using -o 128 -a 128 just makes the underlying sphere smoother, the texture detail remains largely the same. For displacement mapping, with -o 16 -a 16 there aren't enough vertices to accurately represent the complex details of the height map. The physical displacement is blocky and poorly defined because it's "undersampling" the texture. With -o 128 -a 128, there are way more vertices available to be individually pushed and pulled. This allows the displaced geometry to smoothly and accurately match the fine details of the high-resolution texture map.

I added several effects in my custom shader. First, I pushed the vertices of the cloth outward along their normal vectors based on sine and cosine functions of the original x and y coordinates. This physically warps the cloth into a rippling, wavy shape without needing any external height map images. Next, I added a "glowing" effect to the cloth. By taking the dot product of the surface normal and the camera's view vector, I was able to determine the angle at which the user is looking at the surface. I inverted this value so that the center of the cloth (facing the camera dead-on) is a dark, translucent base color, while the grazing angles at the edges transition into a bright, glowing cyan rim. Lastly, I implemented a color-splitting prism effect to give the glass a thick, refractive quality. Instead of sampling the environment cubemap with a single, mathematically perfect reflection vector, I created three slightly offset reflection vectors by intentionally warping the surface normal. I sampled the Red, Green, and Blue channels of the environment map independently using these three different vectors. When combined, this creates a realistic chromatic aberration effect, making the reflections warp and split into different colors just like real curved glass.

Idea 1: Wind

To simulate realistic wind, I implemented an aerodynamic drag model that calculates force based on the cloth's exposed surface area rather than simply applying a constant directional vector to individual point masses. By dividing each two-by-two grid of point masses into two distinct triangles, I calculated their surface normals by taking the cross product of two of their adjacent edge vectors. Because the magnitude of this cross product is equal to twice the triangle's area, it elegantly factors both the facing direction and the physical surface area nto the standard aerodynamic drag equation. Utilizing the dot product between the wind velocity and the surface normal ensures that the wind dynamically exerts maximum force when hitting the cloth dead-on, and zero force when blowing perfectly parallel across it. Finally, this calculated force is distributed equally among the three point masses that make up each triangle, allowing the entire cloth mesh to billow and ripple realistically based on its angle to the wind. To integrate interactive wind controls into the simulation's GUI, I modified three key files to bridge the interface with the physics engine. First, in cloth.h, I added a Vector3D wind_velocity variable to the ClothParameters struct so the simulation state could permanently store the current wind vector. Next, in cloth.cpp, I updated my Cloth::simulate function to pull this dynamic vector directly from the parameters struct rather than using hardcoded values. Finally, in clothSimulator.cpp, I expanded the NanoGUI setup by duplicating the existing Gravity interface block. I relabeled this new section to "Wind" and updated the lambda callback functions on the X, Y, and Z float boxes so that any user input immediately updates the wind_velocity inside the ClothParameters struct. This setup allows the aerodynamic forces to be manipulated and visualized in real-time without recompiling.

Idea 2: Sphere movement

To enable interactive object movement, I capitalized on the existing collision detection architecture. The simulation's collide function dynamically calculates the distance between every cloth point mass and the sphere's origin on every single frame. Because of this continuous evaluation, I realized that physically moving the sphere wouldn't require writing any new physics algorithms; I simply needed a way to update the sphere's center coordinates in real time. Whenever the sphere's position updates, the existing collision logic instantly detects the newly intersecting point masses and pushes them outward to the new surface boundary. This allows the sphere to realistically drag, slide against, and push the draped cloth around the scene purely through the standard collision response. Integrating the GUI controls for the sphere required minor modifications to two files. First, in collision/sphere.h, I altered the encapsulation of the Vector3D origin variable, moving it from the private to the public scope so external interface elements could directly read and modify its coordinates. Next, in clothSimulator.cpp, I updated the initGUI function to iterate through the simulation's collision_objects array and identify the active sphere using a dynamic_cast. Once successfully located, I generated a new "Move Sphere" panel containing X, Y, and Z float box sliders, linking their lambda callback functions directly to the target sphere's origin variables. This ensures that any user adjustment to the sliders instantly translates the sphere in 3D space.

Idea 3: Adding other primitives (Box and Cylinder)

To expand the simulation's collision capabilities, I implemented two new 3D primitives: an axis-aligned Box and a Cylinder. For the Box, the collide function evaluates whether a point mass's current position falls strictly within the object's minimum and maximum 3D coordinates. If an intersection is detected, it calculates the distance to all six faces, identifies the absolute closest face, and pushes the point mass directly outward to that surface. The Cylinder implementation utilizes a hybrid mathematical approach, checking if a point mass is simultaneously within the vertical height limits and the circular radial boundary in the XZ plane. Upon a collision, it calculates the distance to the top cap, bottom cap, and the curved wall, determining the shortest escape path and applying a correction vector to push the point mass safely to the exterior.
